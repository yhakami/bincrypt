# Security Policy

## Supported Versions

| Version | Supported |
|---------|-----------|
| `main`  | ✅ |
| Tagged releases | ✅ |

Report issues against the most recent commit or release. Older, unmaintained forks are out of scope.

## Reporting a Vulnerability

1. **Do not open a public issue or pull request.**
2. Contact the maintainers via email at **security@bincrypt.io** or submit a private GitHub Security Advisory.
3. Include:
   - A detailed description of the vulnerability.
   - Steps to reproduce or proof-of-concept.
   - The impact and any known mitigations.
   - Whether the issue has been disclosed elsewhere.
4. You will receive an acknowledgment within 3 business days. We aim to provide a remediation timeline within 10 business days.

### Coordinated Disclosure

We request a 30-day embargo period to investigate, develop, and ship a fix before any public disclosure. If a wider impact is suspected, we may ask for additional time to coordinate with downstream operators.

## Responsible Deployment

BinCrypt is security‑sensitive software. **No independent third‑party security audit has been completed yet.** Please treat the project accordingly and perform your own review before relying on it for high‑stakes use.

If you operate a modified version of BinCrypt over a network, you must comply with the project license. The open‑source version is AGPL‑3.0; commercial Apache‑2.0 licenses are available for private or closed deployments. See `LICENSE` and the README for details.

## Hardening Checklist

- Keep dependencies up to date (`go get -u` for Go modules, refresh front-end CDNs/SRI hashes).
- Configure production-ready Firebase Storage rules instead of the permissive emulator defaults.
- Protect secrets with Google Secret Manager (or equivalent) and rotate them regularly.
- Enforce HTTPS/TLS, robust rate limiting, and Webhook signature verification when enabling BTCPay.
- Set `METRICS_TOKEN` before enabling `METRICS_ENABLED=true`.
- Monitor audit logs generated by `src/logger` and set alerts for anomalous events.

If you ship BinCrypt commercially, consider a formal security review before each release. Feel free to share high‑level findings with the maintainers so we can further strengthen the project.
